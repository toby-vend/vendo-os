**Create `web/routes/task-runs-ui.ts`** — a `FastifyPluginAsync` exported as `taskRunsUiRoutes`.

Routes to implement:

1. **GET /** — Task list page.
   - Parse query params: `status`, `clientId`, `channel`, `dateFrom`, `dateTo`, `page` (default 1).
   - Call `listTaskRuns({ status, clientId, channel, dateFrom, dateTo, limit: 50, offset: (page-1)*50 })`.
   - Call `listBrandClients()` for client filter dropdown.
   - If `request.headers['hx-request']` is truthy, render `task-runs/list-rows` partial only.
   - Otherwise render `task-runs/list` full page with `{ runs, clients, query: parsedFilters, page, totalPages }`.
   - Note: `totalPages` requires a count query. Add a `countTaskRuns` helper (same WHERE as `listTaskRuns` but `SELECT COUNT(*)`) or estimate from result length. Simplest: if results.length === limit, there are more pages.

2. **GET /rows** — HTMX polling partial (alias for the same logic as GET / with hx-request forced).
   - Same filter parsing as GET /.
   - Always renders `task-runs/list-rows` partial (never full page).

3. **GET /new** — Task submission form page.
   - Call `listBrandClients()` for client dropdown.
   - Render `task-runs/new` with `{ clients }`.

4. **GET /task-types** — HTMX partial returning `<select>` options for task types filtered by channel.
   - Read `channel` from query params.
   - Import and read REGISTRY keys from `web/lib/task-types/index.ts` to get valid taskType values for that channel. The REGISTRY is a Map — iterate its keys, filter those starting with `${channel}:`, extract the taskType part after the colon.
   - Return an HTML `<select name="taskType">` with `<option>` for each valid task type. If channel is empty/invalid, return a disabled select with "Select channel first".
   - NOTE: REGISTRY is not exported. Either export it, or export a helper `getTaskTypesForChannel(channel: string): string[]` from `web/lib/task-types/index.ts`. Prefer the helper — cleaner API. The helper should return the taskType portion (e.g. `['ad_copy']` for `paid_social`).

5. **POST /new** — Form submission handler.
   - Parse form body: `clientId`, `channel`, `taskType`, `instructions` (optional textarea).
   - Validate: clientId is positive integer, channel is valid, taskType is non-empty string.
   - On validation failure, redirect back to `/tasks/new?error=...`.
   - Call `createTaskRun({ clientId, channel, taskType, createdBy: request.user.email })`.
   - Send redirect to `/tasks/${taskRunId}` BEFORE firing assembleContext.
   - Fire-and-forget: `assembleContext(taskRunId, clientId, channel, taskType, request.user.id).catch(...)`.
   - Note: `instructions` field is captured but not stored (no column exists — per research open question, this is acceptable for v1). If the user filled it in, log it but don't block.

6. **Register in `web/server.ts`**:
   - Import `taskRunsUiRoutes` from `./routes/task-runs-ui.js`.
   - Register: `app.register(taskRunsUiRoutes, { prefix: '/tasks' })`.
   - This replaces the old `tasksRoutes` registration which Plan 01 moved to `/asana-tasks`.

**Add routes to `web/routes/task-runs-ui.ts`** (same file from Task 1):

1. **GET /:id** — Draft review / task detail page.
   - Parse `id` from params. Validate positive integer.
   - Call `getAuditRecord(id)` — returns `AuditRecord` with parsed `sops_used: SopSnapshot[]`.
   - If not found, return 404.
   - Parse `run.output` (JSON string) into typed object if status is `draft_ready` or `approved` or `rejected`. Guard for null output.
   - Parse `run.qa_critique` (JSON string) into object for QA/AHPRA banners. Guard for null.
   - Determine which draft partial to use based on `run.channel + ':' + run.task_type`: `ad_copy` -> `task-runs/partials/draft-ad-copy`, `content_brief` -> `task-runs/partials/draft-content-brief`, `rsa_copy` -> `task-runs/partials/draft-rsa-copy`.
   - Render `task-runs/detail` with `{ run, output, sopsUsed: run.sops_used, qaCritique, draftPartial, error: query.error }`.

2. **POST /:id/approve** — Approve a draft.
   - Validate id. Get run via `getTaskRun(id)`.
   - Guard: run must exist and `run.status === 'draft_ready'`. If not, redirect with error.
   - Call `updateTaskRunStatus(id, 'approved')`.
   - Merge `am_feedback: { action: 'approve' }` into existing qa_critique JSON (read, parse, spread, re-serialise). Call `updateTaskRunQA` with merged critique.
   - Redirect to `/tasks/${id}`.

3. **POST /:id/reject** — Reject a draft (requires reason).
   - Parse `reason` from form body. If empty/missing, redirect to `/tasks/${id}?error=reason_required`.
   - Get run, guard status === 'draft_ready'.
   - Read existing `qa_critique`, parse, merge `am_feedback: { action: 'reject', reason: reason.trim() }`, re-serialise.
   - Call `updateTaskRunQA(id, { score: run.qa_score ?? 0, critique: merged })`.
   - Call `updateTaskRunStatus(id, 'rejected')`.
   - Redirect to `/tasks`.

4. **POST /:id/regenerate** — Regenerate (optional comment).
   - Parse optional `comment` from form body.
   - Get run, guard status is `draft_ready` or `rejected` or `failed`.
   - Merge `am_feedback: { action: 'regenerate', comment: comment?.trim() || undefined }` into qa_critique.
   - Call `updateTaskRunQA` with merged critique.
   - Call `updateTaskRunStatus(id, 'queued')` — resets to queued.
   - Send redirect FIRST: `reply.redirect('/tasks/${id}')`.
   - THEN fire-and-forget: `assembleContext(id, run.client_id, run.channel, run.task_type, request.user.id).catch(...)`.
