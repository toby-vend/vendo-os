<feature>
  <name>Skills FTS5 Search, Gap Detection, and Version Tracking</name>
  <files>web/lib/queries/drive.ts, web/lib/queries/drive.test.ts</files>
  <behavior>
    **Search behaviour:**
    - searchSkills("ad copy", "paid_social") returns skills matching "ad copy" in the paid_social channel OR the general channel, ranked by BM25 score
    - searchSkills("ad copy", "paid_social", 3) returns at most 3 results
    - searchSkills("xyznonexistent", "paid_social") returns { results: [], gap: true }
    - searchSkills("template", "paid_social") with results returns { results: [...], gap: false }
    - Search sanitises FTS5 operators (quotes, special chars) before querying — same pattern as meetings search
    - Results include: id, title, content (full), channel, skill_type, drive_modified_at, content_hash, bm25_score

    **FTS5 sync behaviour:**
    - syncSkillFts(rowid, title, content) deletes old FTS entry then inserts new one (content-sync table pattern)
    - deleteSkillFts(rowid) removes the FTS entry
    - Both use the explicit DELETE/INSERT pattern required by content='skills' FTS5 tables

    **Version tracking behaviour:**
    - getSkillVersion("drive_file_id_123") returns { drive_modified_at, content_hash, indexed_at, version } or null
    - getSkillsByVersion("paid_social", sinceDate) returns skills updated after that date in the given channel

    **Gap detection:**
    - When zero results returned, gap is true
    - Gap signal includes query and channel for diagnostic logging
  </behavior>
  <implementation>
    Add to web/lib/queries/drive.ts:

    1. **New interfaces:**
       - `SkillSearchResult`: id, title, content, channel, skill_type, drive_modified_at, content_hash, bm25_score (number)
       - `SkillSearchResponse`: { results: SkillSearchResult[], gap: boolean, query: string, channel: string }
       - `SkillVersionInfo`: { drive_modified_at: string, content_hash: string, indexed_at: string, version: number }

    2. **searchSkills(query: string, channel: string, limit = 5): Promise<SkillSearchResponse>**
       - Sanitise query: strip quotes, split on whitespace, append `*` to each token, join with space (same as meetings)
       - If sanitised query is empty, return { results: [], gap: true, query, channel }
       - SQL: SELECT s.id, s.title, s.content, s.channel, s.skill_type, s.drive_modified_at, s.content_hash, bm25(skills_fts) as bm25_score FROM skills_fts fts JOIN skills s ON s.rowid = fts.rowid WHERE skills_fts MATCH ? AND (s.channel = ? OR s.channel = 'general') ORDER BY bm25(skills_fts) LIMIT ?
       - Note: bm25() returns negative values (more negative = more relevant) — order ASC
       - Return { results, gap: results.length === 0, query, channel }

    3. **syncSkillFts(rowid: number, title: string, content: string): Promise<void>**
       - Execute: INSERT INTO skills_fts(skills_fts, rowid, title, content) VALUES('delete', ?, ?, ?)
       - Then: INSERT INTO skills_fts(rowid, title, content) VALUES(?, ?, ?)
       - Use db.batch() or sequential execute for atomicity

    4. **deleteSkillFts(rowid: number, title: string, content: string): Promise<void>**
       - Execute: INSERT INTO skills_fts(skills_fts, rowid, title, content) VALUES('delete', ?, ?, ?)
       - Must pass the OLD title and content values for the delete command to work with content-sync tables

    5. **getSkillVersion(driveFileId: string): Promise<SkillVersionInfo | null>**
       - SELECT drive_modified_at, content_hash, indexed_at, version FROM skills WHERE drive_file_id = ?

    6. **getSkillsByVersion(channel: string, since: string): Promise<SkillRow[]>**
       - SELECT * FROM skills WHERE channel = ? AND indexed_at > ? ORDER BY indexed_at DESC

    **Test setup:**
    - Use Node test runner (node:test) with --experimental-test-module-mocks flag (established pattern from Phase 2)
    - Tests need a real SQLite database since FTS5 queries must execute against a real engine
    - Create an in-memory libsql database, run schema creation (skills table + skills_fts), insert test fixtures, run search queries
    - If libsql in-memory is not feasible for tests, mock at the query level and test the SQL construction + result mapping separately
  </implementation>
