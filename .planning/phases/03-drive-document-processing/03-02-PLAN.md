**processChange(change: DriveChange, accessToken: string): Promise<void>**
Follow Pattern 4 from RESEARCH.md exactly:
1. If `changeType !== 'file'` return (ignore drive-level changes).
2. If `change.removed || change.file?.trashed` -> call `deleteSkill(fileId)` and return.
3. If no `change.file` return.
4. Call `resolveChannel(fileId, accessToken)`. If null -> `deleteSkill(fileId)` and return (moved outside watched folders).
5. Call `resolveSkillType(fileId, accessToken)`.
6. Call `extractContent(fileId, mimeType, accessToken)`.
7. If content is not null:
   a. Compute `hashContent(content)`.
   b. Get existing skill via `getSkillByDriveFileId(fileId)`.
   c. If existing hash matches -> call `updateSkillMetadata` (rename/reclassify only).
   d. Else -> call `updateSkillContent` (full upsert with content + hash).
8. If content is null -> call `updateSkillMetadata` (non-indexable type, metadata only).

**processQueue(): Promise<{ processed: number; errors: number }>**
Follow Pattern 1 from RESEARCH.md:
1. Get unprocessed queue items (limit 50).
2. If empty, return `{ processed: 0, errors: 0 }`.
3. Group by `channel_id` (deduplicate — multiple webhooks may fire for same batch).
4. For each unique channel_id:
   a. Get channel record via `getDriveWatchChannel`.
   b. Skip if no channel or no page_token.
   c. Get userId from channel.user_id or `process.env.DRIVE_ADMIN_USER_ID`.
   d. Call `listChanges(userId, pageToken)`.
   e. **Persist new pageToken immediately** via `updateDrivePageToken` (before processing — per Research pitfall 2).
   f. For each change, call `processChange` wrapped in try/catch. Count errors.
5. Mark all queue items as processed via `markQueueItemProcessed` (with error string if failed).
6. Return `{ processed, errors }`.

**CLI entry point** (runs when executed directly):

async function main() {
  const result = await processQueue();
  console.log(`Processed: ${result.processed}, Errors: ${result.errors}`);

In the file processing loop (where `upsertSkillFromDrive` is called), replace the upsert logic:

For each file with an indexable MIME type:
1. Call `extractContent(file.id, file.mimeType, accessToken)`.
2. If content returned, compute `hashContent(content)`.
3. Derive `skillType` — since re-index walks from known channel folders, the parent folder structure is already known. Use a simplified approach: if the file's immediate parent is the top-level channel folder, `skillType = 'sop'`. Otherwise, pass the subfolder name through the slugify logic. The `resolveSkillType` helper requires API calls which are wasteful during re-index (we already know the structure). Instead, track the subfolder name during the recursive walk by adding a parameter to `listFilesInFolder` (or a wrapper) that records the first subfolder under the channel root.
4. Call `updateSkillContent({ driveFileId, title, content, contentHash, channel, skillType, driveModifiedAt })` instead of `upsertSkillFromDrive`.
5. For files where `extractContent` returns null (non-indexable), continue using `upsertSkillFromDrive` for metadata-only upsert.
