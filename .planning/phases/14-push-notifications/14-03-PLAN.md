**1. Toast banner in base.eta:**

Add a script block in `web/views/layouts/base.eta` (inside the existing script area, after SW registration code) that implements the push permission toast banner. The logic:

a. **Helper functions** (inline, no external deps):
```javascript
function urlBase64ToUint8Array(base64String) {
  var padding = '='.repeat((4 - base64String.length % 4) % 4);
  var base64 = (base64String + padding).replace(/-/g, '+').replace(/_/g, '/');
  var rawData = atob(base64);
  return Uint8Array.from([...rawData].map(function(c) { return c.charCodeAt(0); }));
}

function isIOS() { return /iphone|ipad|ipod/i.test(navigator.userAgent); }
function isStandalone() {
  return window.matchMedia('(display-mode: standalone)').matches || window.navigator.standalone === true;
}
```

b. **Toast banner display logic** — triggered by a custom event `vendoos:task-completed` (dispatched when HTMX swaps task completion responses). Also check on page load if the URL contains a task completion indicator.

The banner appears if ALL conditions are met:
- `'serviceWorker' in navigator` (SW support)
- `'PushManager' in window` (Push API support)
- `Notification.permission !== 'granted'` (not already subscribed)
- `localStorage.getItem('push-toast-dismissed') !== '2'` (not dismissed twice)

c. **Banner content depends on state:**
- If iOS and NOT standalone: text = "To receive notifications, install VendoOS to your home screen first." with a link `<a href="/settings#install-app-section">Install App</a>`. No enable button.
- If `Notification.permission === 'denied'`: text = "Notifications are blocked. To enable, open your browser settings for this site." No button — informational only.
- Otherwise (normal flow): text = "Get notified when drafts are ready —" with an `[Enable notifications]` button.

d. **Enable button click handler:**
- Call `await Notification.requestPermission()` (Promise form, not callback).
- If `'granted'`: fetch `GET /api/push/vapid-public-key`, get the key, call `navigator.serviceWorker.ready`, then `registration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: urlBase64ToUint8Array(key) })`, then `POST /api/push/subscribe` with `sub.toJSON()`.
- Remove the toast banner on success.

e. **Dismiss behaviour:**
- Banner has an X close button.
- On dismiss: increment `localStorage` counter (`push-toast-dismiss-count`). If count reaches 2, set `push-toast-dismissed` to `'2'` (never show again per locked decision).
- If dismissed once, show again on next task completion event.

f. **Task completion detection:**
Listen for HTMX `htmx:afterSwap` events. Check if the swapped content indicates a task status change to `draft_ready` or if the page is a task detail page showing draft_ready status. Alternatively, use a simpler approach: dispatch `vendoos:task-completed` from task-runs-ui routes that return draft_ready status, or add a `data-task-completed` attribute to the response HTML.

The simplest reliable approach: on page load, check if `window.location.pathname.startsWith('/tasks/')` and the page contains a `draft_ready` badge. This covers the "after first task completes" trigger without requiring server-side changes.
