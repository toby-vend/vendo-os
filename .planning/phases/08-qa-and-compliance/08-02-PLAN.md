    **B) Modify `web/lib/task-matcher.ts` generateDraft:**

    Replace the current `generateDraft` function with a QA-aware version:

    1. Change `MAX_ATTEMPTS` from 2 to 3 (initial + 2 QA retries). The inner retry for API/parse errors is removed — failures now throw and the outer loop handles them.
    2. The loop structure becomes:
       ```
       for attempt 0..2:
         - Build user message (on retry, prepend critique: "Previous attempt failed QA. Issues: {critique}. Please regenerate addressing these issues.")
         - Same system prompt on every attempt (locked decision)
         - Call Anthropic (Sonnet) — on API/parse error, throw (assembleContext catch handles transition to failed)
         - Parse JSON, validate sources array
         - Transition status to qa_check: await updateTaskRunStatus(taskRunId, 'qa_check')
         - Increment attempts: await incrementAttempts(taskRunId)
         - Call runSOPCheck(JSON.stringify(parsed), sopContent)
         - If pass:
           - Run checkAHPRACompliance(JSON.stringify(parsed))
           - Build critique: { sop_issues: [], ahpra_violations: ahpraResult }
           - await updateTaskRunQA(taskRunId, { score: 1, critique: JSON.stringify(critiqueObj) })
           - await updateTaskRunOutput(taskRunId, JSON.stringify(parsed))
           - return
         - If fail and more attempts:
           - Store critique for next iteration
           - Continue loop
         - If fail and no more attempts:
           - Run checkAHPRACompliance(JSON.stringify(parsed))
           - Build critique: { sop_issues: sopIssues, ahpra_violations: ahpraResult }
           - await updateTaskRunQA(taskRunId, { score: 0, critique: JSON.stringify(critiqueObj) })
           - await updateTaskRunOutput(taskRunId, JSON.stringify(parsed))
           - return
       ```
    3. Add `buildRetryMessage` helper (private): takes the original user message content, prepends the critique. Format: `"Previous attempt failed QA. Issues:\n{critique}\n\nPlease regenerate addressing these issues.\n\n{originalUserMessage}"`

    6. Wrap the QA section in try/catch — on unexpected QA error, transition to `failed` (not stuck at qa_check). Same pattern as existing error handling.
