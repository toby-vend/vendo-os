**1. Page-level swipe navigation (research Pattern 4):**

Implement a self-invoking function that:
- Defines `SECTIONS = ['/', '/tasks', '/clients']`
- Gets the current path via `window.location.pathname`
- On `touchstart` (passive): records `startX`, `startY`, `startTime`
- Checks `e.target.closest('[data-no-swipe], .task-card-wrapper, .overflow-x-scroll, .portal-date-pills')` — if truthy, bails out (prevents conflicts with card swipe and horizontal scrollable elements per research Pitfall 6)
- On `touchend` (passive): calculates dx, dy, dt
- Only triggers if `Math.abs(dx) > 60`, `Math.abs(dx) > Math.abs(dy) * 1.5`, and `dt < 400`
- Finds current section index; `dx < 0` (swipe left) goes to next section, `dx > 0` (swipe right) goes to previous
- Navigates via `window.location.href = next`

**2. Task card swipe actions (research Pattern 6):**

Implement a function `initTaskCardSwipes()` that:
- Queries all `.task-card-wrapper` elements
- For each wrapper, gets the `.task-card` inside and reads `data-task-id` and `data-status`
- On `touchstart` (passive) on the wrapper: records startX
- On `touchend` (passive) on the wrapper:
  - dx > 60 and status === 'draft_ready': toggles `.swiped-right` on the `.task-card`, shows the `.swipe-action-approve` div (`style.display = 'flex'`), and uses `e.stopPropagation()` — **note:** this requires a non-passive touchend listener on the wrapper specifically (remove passive for touchend on card wrappers to allow stopPropagation)
  - dx < -60: navigates to `/tasks/{taskId}`
  - Tapping the `.swipe-action-approve` button: fires `POST /api/tasks/runs/{taskId}/approve` via `fetch()`, then reloads the card list by triggering `htmx.trigger(document.getElementById('task-rows'), 'refresh')`. Show a brief "Approved" flash on the card before refresh.

Call `initTaskCardSwipes()` on DOMContentLoaded. Also re-init after HTMX swaps on `#task-rows` (add `document.addEventListener('htmx:afterSettle', function(e) { if (e.detail.target.id === 'task-rows') initTaskCardSwipes(); })`).

**Important:** The approve POST endpoint. Check whether `/api/tasks/runs/:id/approve` exists. If the existing approval flow is via the detail page form, the swipe approve should `POST` to the same endpoint the detail page uses. Read `web/routes/tasks.ts` to find the approve endpoint URL and method. Use that exact URL.

**3. Pull-to-refresh (research Pattern 5):**

Implement a self-invoking function that:
- Only activates if `document.getElementById('task-rows')` exists (task list page only)
- Tracks `touchstart` on `.main-content` when `scrollTop === 0`
- On `touchmove` (passive): calculates pull distance. If > 20px, shows a `.ptr-indicator` element (create it dynamically: a small div with a rotating SVG spinner, prepended to `.main-content`)
- On `touchend`: if pull distance > 70px, triggers `htmx.trigger(document.getElementById('task-rows'), 'refresh')` and shows the spinner briefly. Listens for `htmx:afterSettle` on `#task-rows` to remove the spinner.
- If pull distance < 70px, removes the indicator immediately.
- The pull indicator div has class `.ptr-indicator` and contains a small SVG circular arrow icon.

**4. Pull-to-refresh indicator CSS (append to Phase 11 section in style.css):**

Inside the 768px media query:
- `.ptr-indicator`: `display: flex; align-items: center; justify-content: center; padding: 0.75rem; color: var(--vendo-text-muted);`
- `.ptr-indicator svg`: `width: 20px; height: 20px; animation: ptr-spin 0.8s linear infinite;`
- `@keyframes ptr-spin { to { transform: rotate(360deg); } }` (this goes outside the media query since keyframes are global)

**5. Reset swiped cards on tap elsewhere:**

Add a document click/touchstart listener that resets any `.task-card.swiped-right` back to normal (remove class, hide approve button) when tapping outside a card wrapper. This prevents cards from staying in swiped state.
